# Record Schema

A xists record is an automatically generated description of one repository.

Records are used as the source data for future search, ranking, LLM profiling, and result display. A record should be traceable: readers should be able to understand where the information came from, when it was collected, and which parts are missing or uncertain.

## Principles

- Records are generated by xists, not hand-written.
- Factual fields come from source APIs such as the GitHub API.
- Derived fields must be based on collected source data.
- LLM-generated fields, when added, must be based on collected evidence.
- Missing or incomplete information should be recorded explicitly in `evidence_gaps`.
- Records should keep enough metadata to explain when and how they were produced.

## Versioning

Each record includes:

```json
"schema_version": 1
```

`schema_version` is the integer version of the xists record schema.

Current version: `1`.

Increase this number only when the record structure changes in a way that older xists code may not understand correctly.

Examples of changes that should increase `schema_version`:

- Renaming an existing field.
- Removing an existing field.
- Changing the type of an existing field.
- Changing the meaning of an existing field.
- Moving fields into a different structure in a breaking way.

Examples of changes that do not necessarily require a new schema version:

- Adding a new optional field.
- Adding a new evidence kind.
- Adding a new structure signal.
- Adding more metadata while keeping existing fields compatible.

## Traceability

Each record should include traceability fields:

```json
{
  "snapshot_source": "github_api",
  "snapshot_time": "2026-06-19T10:37:12.345678+00:00"
}
```

`snapshot_source` identifies where the source snapshot came from. For GitHub ingestion, this is currently `github_api`.

`snapshot_time` is the UTC time when xists generated the record from the collected source data. It is not the repository creation time, update time, or push time.

This field is useful for:

- Knowing when the record was collected.
- Deciding when a record should be refreshed.
- Explaining why a record may differ from the current GitHub page.
- Comparing old and new records during future incremental updates.

## Top-level fields

A version 1 record currently has this shape:

```json
{
  "schema_version": 1,
  "repo_id_requested": "facebook/react",
  "repo_id": "react/react",
  "platform": "github",
  "owner": "react",
  "name": "react",
  "url": "https://github.com/react/react",
  "github": {},
  "readme": {},
  "structure": {},
  "evidence": [],
  "evidence_gaps": [],
  "lifecycle_state": "candidate",
  "snapshot_source": "github_api",
  "snapshot_time": "..."
}
```

## Repository identity

### `repo_id_requested`

The repository identifier requested by the user or input file.

Example:

```json
"repo_id_requested": "facebook/react"
```

This field is preserved even if GitHub redirects the repository.

### `repo_id`

The canonical repository identifier returned by the source platform.

Example:

```json
"repo_id": "react/react"
```

This can differ from `repo_id_requested` when a repository has been renamed or transferred.

### `platform`

The source platform.

Current value:

```json
"platform": "github"
```

### `owner`, `name`, `url`

Basic repository identity fields from the source platform.

## GitHub fields

The `github` object contains factual metadata collected from the GitHub API.

Example:

```json
{
  "description": "The library for web and native user interfaces.",
  "topics": ["declarative", "frontend", "javascript", "library", "react", "ui"],
  "stars": 245995,
  "forks": 49277,
  "language": "JavaScript",
  "license": "MIT",
  "archived": false,
  "disabled": false,
  "homepage": "https://react.dev",
  "default_branch": "main",
  "created_at": "2013-05-24T16:15:54Z",
  "updated_at": "...",
  "pushed_at": "...",
  "open_issues": 0
}
```

These values are collected from GitHub. They should not be hand-written.

## README fields

The `readme` object contains README information collected from the GitHub README API.

Example:

```json
{
  "path": "README.md",
  "source_url": "https://github.com/react/react/blob/main/README.md",
  "download_url": "https://raw.githubusercontent.com/react/react/main/README.md",
  "excerpt": "React is a JavaScript library for building user interfaces..."
}
```

`excerpt` is a cleaned text excerpt from the README. It may remove empty lines, badge lines, and simple Markdown decoration lines.

If no README is available, `readme` is `null`.

## Structure fields

The `structure` object contains information derived from the repository file tree.

Example:

```json
{
  "signals": [
    "has_readme",
    "has_package_json",
    "has_js_lockfile",
    "has_packages_directory",
    "has_examples_or_fixtures",
    "has_tests",
    "has_scripts_directory"
  ],
  "tree_file_count": 7891,
  "tree_truncated": false
}
```

`signals` are rule-based indicators derived from repository paths. They are not LLM-generated.

Current examples include:

- `has_readme`
- `has_package_json`
- `has_python_manifest`
- `has_go_mod`
- `has_cargo_manifest`
- `has_js_lockfile`
- `has_src_directory`
- `has_packages_directory`
- `has_docs_directory`
- `has_examples_or_fixtures`
- `has_tests`
- `has_scripts_directory`

## Evidence

The `evidence` array contains source-backed evidence that explains what information supports the record.

Example:

```json
[
  {
    "kind": "github_description",
    "text": "The library for web and native user interfaces.",
    "source_url": "https://github.com/react/react"
  },
  {
    "kind": "github_topics",
    "text": "declarative, frontend, javascript, library, react, ui",
    "source_url": "https://github.com/react/react/topics"
  }
]
```

Current evidence kinds include:

- `github_description`
- `github_topics`
- `readme_excerpt`
- `structure_signals`

Evidence should be traceable to a real source when possible.

## Evidence gaps

The `evidence_gaps` array records missing or incomplete information.

Example:

```json
[
  "missing_github_description",
  "missing_github_topics",
  "missing_readme_excerpt"
]
```

Current gap examples include:

- `missing_github_description`
- `missing_github_topics`
- `missing_readme_excerpt`
- `missing_repository_tree`
- `truncated_repository_tree`
- `missing_tree_paths`

Gaps are important because xists should prefer abstaining or lowering confidence when evidence is insufficient.

## Lifecycle state

Each record has a lifecycle state.

Current default:

```json
"lifecycle_state": "candidate"
```

Initial states:

- `candidate`: automatically generated and not yet human-verified.
- `verified`: reviewed or confirmed by a human or future verification process.
- `excluded`: intentionally excluded from search or recommendation.

## Future LLM profile

LLM-generated fields should be added separately from collected GitHub facts.

A future shape may look like:

```json
{
  "llm_profile": {
    "provider": "anthropic",
    "model": "claude-sonnet-4-6",
    "generated_at": "2026-06-19T10:40:00+00:00",
    "input_evidence_kinds": [
      "github_description",
      "github_topics",
      "readme_excerpt",
      "structure_signals"
    ],
    "summary": "...",
    "use_cases": [],
    "capabilities": [],
    "not_for": [],
    "search_phrases": []
  }
}
```

LLM profile generation should not replace collected source fields. It should add interpretation on top of traceable evidence.
